01 — Overview

Introduction to Database Normalization

Normalization is an important process in database design that helps improve the database's efficiency, consistency, and accuracy. It makes data easier to manage and maintain, and keeps the database adaptable to changing business needs.

It organizes attributes to reduce or eliminate data redundancy — the same data stored in different places, which unnecessarily increases the size of the database.

Data redundancy causes inconsistency during insert, delete and update operations, making data management difficult. Normalization splits a table into multiple tables, which are linked whenever a query needs their combined data.

03 — Motivation

The Need for Normalization

The primary goal of normalizing relations is to eliminate three anomalies. Left unchecked, they cause data redundancy and threaten data integrity as a database grows.

Insertion Anomalies

You can't insert data because required fields are missing or incomplete — e.g. a record can't be added without a value for its primary key.

Deletion Anomalies

Deleting one record unintentionally destroys other data — e.g. removing a customer also wipes out every order linked to them.

Update Anomalies

Updating data in one place but not every duplicate copy — e.g. an employee's salary changes in one record but not the others, breaking reports.

04 — Benefits

Features of Database Normalization

Eliminates Redundancy

Removes repeated data across the database, improving efficiency and consistency.

Ensures Consistency

Prevents contradictions that arise from multiple versions of the same fact.

Simplifies Management

Breaks complex structures into simpler tables that are easier to update and retrieve.

Improves Design

Structured, systematic organization that's easier to maintain and adapt over time.

Standardization

Data is stored in a consistent, uniform manner via well-defined table relationships.

1NF

First Normal Form

A table is in 1NF if:

  • All attributes contain only atomic (indivisible) values.
  • Each column contains values of a single type.
  • Each row is unique and identifiable by a primary key.

2NF

Second Normal Form

  • Meet 1NF requirements — all columns hold single, indivisible values with no repeating groups.
  • Eliminate partial dependencies — a non-prime attribute must not depend on only part of a composite primary key.

A functional dependency A → B is a partial dependency if B depends on A, but B can also be determined by a proper subset of A. With a composite key made of more than one attribute, if some attribute depends on only part of that key rather than the whole key, that's a partial dependency — exactly what 2NF forbids.

3NF

Third Normal Form

A relation is in 3NF if it satisfies both:

  • It is already in 2NF — no partial dependencies exist.
  • There is no transitive dependency for non-prime attributes — no non-key attribute depends on another non-key attribute; every non-key attribute must depend directly on the primary key.

BCNF

Boyce–Codd Normal Form

  • The table must already be in 3NF.
  • For every functional dependency X → Y, X must be a super-key.

BCNF prevents update, insert, and delete anomalies by ensuring every determinant is a super key — something 2NF and 3NF can still miss when a dependency's determinant isn't one.

4NF

Fourth Normal Form

  • The relation must already be in BCNF.
  • It must have no multivalued dependency.

A multivalued dependency occurs when one attribute determines multiple independent values of another — it always needs at least three attributes, where two are independent but both depend on a third (A →→ B, with B and C independent).

Fifth Normal Form — Project-Join Normal Form

5NF is one of the highest levels of normalization. A table is in 5NF if it's already in 4NF and it cannot be broken down into smaller tables without losing data.

Rejoining R1 ⋈ R2 produces an extra tuple that never existed in the original data (e.g. science / class 10 / kartik) — a lossy decomposition. That breaks 5NF's rule of non-loss decomposition, so this particular split is not a valid 5NF structure.
